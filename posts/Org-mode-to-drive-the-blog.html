Org mode to drive the blog

Over the last months I have totally switched to org-mode. For me it all started with tracking tasks and todos within emacs. But then I realized that I don't create .txt and .md files anymore, because I started to make my notes in org files. And then I realized that I abandoned my lovely KitsuneBook. Can't say that I am totally happy with it (hey, I've lost a great fellow pet project), but org-mode gave me much more in return. It solved my seemingly never-ending note organization problem, it allowed me to use some neat features right in these regular text files (like references, reminders, code sourcing and editing, tables with formulas, latex, tasks, planning etc). And most important - it suits ideally my emacs-overusing model. This model says - stay in emacs as much as possible. And now, I even generating my blog using org-mode.

When I was creating my blog I was looking for tool that could easily create static blog from bunch of files. I knew that github pages are jakyll-friendly, but I was like 'why so easy?'. So I installed hakyll (from the third time). It's really easy to use in simple cases like mine and easy to extend in more complicated cases. I mean, easy for those who can speak Haskell. So hakyll was a good friend of mine for some time. And I was happy. It was a bit too complicated for my very simple case, but it's written in Haskell, so who cares?

When I was creating my blog I was writing everything in markdown. Recently I found that pandoc (the tool that is used by hakyll to convert markdown files to html) can convert org files to html as well. Well, I wasn't actually surprised, it's pandoc anyway. So I started to write for this blog in org. But what bothered me - is a serious problem with duplicating files. You see, I usually create notes in my working directory (say, in org-home-path or previously, in kitsunebook home). I write them, update them. And when I decide that I want to share this note - I just copy and paste parts that I want to share into new file in blog repository. And that looks fine until I need to change something in shared note. Then I need to make all modifications in my working directory then copy them to blog. And I can't just copy the whole file because

  1. It contains some information that is not public (like some references, todos, draft parts that I am still probably working on). And while it's part of my working directory - it gains all bonuses from it.
  2. I need to add some meta information (like title, tags) that is used by hakyll right into the file. And I hate having this meta information in file that is in my working directory, because it breaks some of my functions that I am lazy to fix. Actually, this is a minor problem. But anyways.

So honestly I started to think about moving out from hakyll to any tool that would allows me to create static blog from org files. I found few tools that looked promising (most interesting from them was o-blog), but it didn't worked for me. Mostly because of complicated configuration and being over complicated. So I looked back at org-mode. I knew that it has some publishing feature, but I never really tried it. And now I am surprised by how it suits my needs.

Making the configs

Leuven making you jealous

Few words before (don't read it)

Whenever I read about org-mode features in manual, I end up googling examples, trying them and then returning to manual. With project publishing situation was pretty similar. I've read publishing section and then was like: 'So how do I configure it?'. Then I found some good examples (on org official site) and I experienced delightful moment of understanding.

So to the configs.

Important notice before we start (sorry for delaying!)

org-mode is a big package. So a lot of stuff is loaded lazily. When you experiment with publishing configurations you might want to (require 'ox-publish). Otherwise you'll face undefined functions until restart.

Structure

My blog (a bit simplified, avoiding all hakyll-related stuff) has following structure: 

.
├── Archive.org # auto-generated file
├── css         # folder where I store my css files
│   └── ...
├── drafts      # folder where I store my drafts
├── images      # folder where I store images used by site and posts
│   └── ...
├── index.org   # index file
├── js          # folder where I store my js files
│   └── ...
└── posts       # folder where I store public posts
    └── ...

It's pretty simple, isn't it?

How to…

In order to publish a blog site from org-mode you need to declare it as your project. Project is defined by specifying it's name and property values. 

("project-name" :property value :property value ...)

But it's also possible to define project as a composition of other projects. 

("project-name" :components ("project-1" "project-2" ...))

So you define several projects and put them into org-publish-project-alist. 

(setq org-publish-project-alist
      '(("project-1" :property value :property value ...)
        ("project-2" :property value :property value ...)
        ("project-name" :components ("project-1" "project-2" ...))))

Back to the garden

So my blog-project has 3 sub-projects:

  1. posts (org files in posts directory)
  2. static files (images, css and js files)
  3. index page

Probably you've noticed that Archive.org file. I didn't mentioned it because it's auto-generated in first project. So let's define it. 

("d12-posts"
 :base-directory "~/Dropbox/org/d12frosted/"               ; directory where org files are stored
 :base-extension "org"                                     ; extensions of files you are going to publish
 :exclude "index.org\\|archive.org"                        ; files to exclude
 :publishing-directory "~/Developer/d12frosted.github.io/" ; directory where to publish
 :publishing-function org-html-publish-to-html             ; function used to publish matched files
 :html-extension "html"                                    ; extension of resulting files
 :recursive t                                              ; recursive search
 :auto-sitemap t                                           ; auto-generate sitemap (archive)
 :sitemap-filename "archive.org"                           ; the name of generated file
 :sitemap-title ""                                         ; title of generated file (read further for more info)
 :sitemap-style list                                       ; default is tree, but we don't want to see directories tree
 :sitemap-sort-files anti-chronologically                  ; newer at the top
 :sitemap-file-entry-format "%d - %t"                      ; date - titile
 )

Pretty straightforward, isn't it?

What is so special about index?

We want it to be published, but we don't want it to be in archive list. I couldn't find a better way for achieving this rather than creating separate project just for index file. 

<Some intro goes here>

*ARCHIVE*

\#+INCLUDE: "archive.org"

Archive.org file is auto-generated. And it's OK. So we need a separate file where we can add additional information (e. g. about ourselves) and also show the list of all posts. Previously I've set sitemap-title to ~""~ because of this including. I don't want to this it's title on index page. 

("d12-index"
 :base-directory "~/Dropbox/org/d12frosted/"               ; directory where index.org is stored
 :base-extension "org"                                     ; the default value
 :exclude "Archive\.org"                                   ; we don't want any special treatment for
 :publishing-directory "~/Developer/d12frosted.github.io/" ; directory where to publish
 :publishing-function org-html-publish-to-html             ; function used to publish matched files
 :html-extension "html"                                    ; extension of resulting files
)

The good thing about this project - you can add About.org file and it will be published as well. Just don't forget to ignore in first project.

What about static files?

As I told, in my case there are three (actually, four) projects. The third one is about static files. And here is a truth - base-extension is actually regexp. So you don't need to create separate projects for every type of static files you want to share. For static files (the files that don't require any conversion / compiling) there is a separate function - org-publish-attachment. Let's see how I configured this project. 

("d12-static"
 :base-directory "~/Dropbox/org/d12frosted/"               ; directory where static files are stored
 :base-extension "png\\|jpg\\|gif\\|css\\|js"              ; extensions of static files
 :publishing-directory "~/Developer/d12frosted.github.io/" ; directory where to move (publish) static files
 :publishing-function org-publish-attachment               ; function to use for publishing
 :recursive t                                              ; recursive search
)

Nothing interesting here.

Combine it till it combines

And the most easiest part. Now we are going to configure a project that combines two previously configured. 

("d12"
 :components ("d12-static"
              "d12-posts"
              "d12-index"))

In my case the parent project is very simple. So again, nothing interesting.

In case you're stuck

To publish current file use C-c C-e P f. To publish current project - C-c C-e P p. To publish all files - C-c C-e P a. But sometimes it will say 'skipping X file, because it hasn't changed'. Know this - rm -r ~/.org-timestamps. You can configure the directory of publish timestamps by changing the value of org-publish-timestamp-directory.

So what?

  1. All my notes / posts are in one place. So I get all bonuses from org-mode and my configurations for it (like agenda, search etc).
  2. I can easily add custom configurations to certain posts (like toc, visibility of different parts, either export todos or not etc).
  3. No duplicates!
  4. Drafts vs Posts. No more agony!

I am still thinking about automating publishing process even more. org-mode allows you to call custom function on pre-publishing and post-publishing. So it's even possible to auto commit and push on successful site generation (with new content).

NB! My configurations are changing very often. In case you are interested in latest changes, checkout org-configs.el!
